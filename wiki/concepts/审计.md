## 概念定义

审计（Audit）是 [[Claude Code Dynamic Workflows]] 的核心组成部分，代表了动态工作流生命周期中的一个关键阶段。在工作流的“规划-审计-执行”模型中，审计阶段位于规划之后、执行之前。其核心目的是赋予用户对即将执行的工作流的完全可见性和控制权。在此阶段，用户可以详细审查系统自动生成的“工作流脚本”（Workflow Script），理解每一个步骤的逻辑、所调用的 [[子智能体]]（Sub-agents）或 [[外部工具]]（External Tools），以及它们将如何协同工作。更重要的是，审计阶段会清晰地展示预计将使用的 [[模型]]（如 [[Opus]]、[[Sonnet]]、[[Haiku]] 等）以及这些模型和智能体在整个工作流执行过程中可能产生的资源消耗（如计算时间、API 调用次数等）。通过这种方式，审计机制能够有效防止意外的资源浪费、不符合预期的行为，并确保工作流的执行符合安全和成本效益的要求。它为用户提供了一个“批准前不执行”的安全网，极大地增强了用户对自动化流程的信任感和掌控力。

## 技术细节

审计阶段的实现依赖于 [[Claude Code Dynamic Workflows]] 的内部状态管理和可视化能力。当用户输入 `ultra code` 指令激活动态工作流后，系统首先进入“规划”阶段，生成一个详细的“执行计划”（Workflow Plan）和流程图。随后，工作流进入“审计”阶段。在此阶段，系统会解析规划阶段生成的执行计划，将其转化为用户可读的格式，包括：

*   **工作流脚本预览：** 展示构成工作流的各个命令、逻辑分支、条件判断等。
*   **模型与智能体配置：** 列出每个任务点将调用的具体模型（例如，用于代码分析的 [[Opus]]，用于数据汇总的 [[Sonnet]]）以及与之关联的 [[子智能体]]。
*   **预计消耗：** 提供对智能体配额、API 调用次数、预估执行时间的量化估算。这有助于用户进行 [[预算控制]]。
*   **[[外部工具]] 集成详情：** 如果工作流集成了 [[CodeGraph MCP]] 或 [[Codex CLI]] 等外部工具，审计阶段会展示这些工具的调用方式和预期作用。

用户可以在此阶段对工作流进行审查，并可以选择“批准”以进入执行阶段，或选择“修改”或“取消”工作流。这种“批准前不执行”的机制是审计阶段的核心价值所在，它确保了用户在投入实际计算资源之前，能够充分理解并认可工作流的潜在影响。

## 应用场景

审计在 [[Claude Code Dynamic Workflows]] 的应用中扮演着至关重要的角色，尤其是在处理复杂、高风险或资源密集型任务时：

*   **大规模代码审查与 Bug 分析：** 在对大型代码库进行自动化审查或分析大量 GitHub Issue 时，审计阶段允许开发者在执行耗时且可能产生大量 API 调用的分析任务前，检查工作流是否正确地识别了关键代码区域或 Issue 优先级。例如，在分析 50 个 Issue 时，用户可以审计工作流是否准确地筛选出了 T10 级别的核心 Bug，以及是否计划使用合适的模型进行 [[对抗验证]]。
*   **资源密集型任务的成本控制：** 对于需要调用旗舰模型（如 [[Opus]]）或进行大规模 [[并发执行]] 的任务，审计阶段能够让用户预估潜在的成本，并根据预算进行调整。这有助于避免因意外的资源消耗而导致成本失控。
*   **安全敏感的代码操作：** 在涉及代码重构、自动化修复或与生产环境交互的任务中，审计阶段提供了一个关键的检查点，确保执行的操作符合预期，避免引入新的 Bug 或数据丢失。
*   **[[多模型协作]] 的验证：** 当工作流设计为利用不同模型的优势（如 [[Opus]] 的深度分析能力与 [[Haiku]] 的速度优势）时，审计阶段可以帮助用户确认模型之间的协作逻辑是否合理，以及预期的 [[互补优势]] 是否能得到有效发挥。

通过审计阶段，用户能够更自信地利用 [[Claude Code Dynamic Workflows]] 来处理从简单的代码导航到复杂的系统级架构分析等各种任务，确保自动化流程的准确性、效率和安全性。
