## Agent 工作方式

Agent 工作方式是指 Claude Code 在其最新的动态工作流（Dynamic Workflows）功能中展现出的核心运作模式。它标志着 Claude Code 从一个传统的单一 AI 助手，进化为一个能够充当“智能体指挥官”的角色。这种工作方式的核心在于其自动化、智能化和协作化的能力，能够自主地规划复杂的任务流程，并根据需要派生出多个专门的子智能体（Sub-agents）来执行任务的不同环节。

在 Agent 工作方式下，Claude Code 不再是简单地响应用户的单一指令，而是能够理解更宏观的目标，并将其分解为一系列可执行的步骤。这些步骤可能涉及调用 [[外部工具]]，例如 [[CodeGraph MCP]] 或 [[Codex CLI]]，以获取和处理外部数据或执行特定的代码操作。更重要的是，它能够 orchestrate（编排）多个智能体之间的协作，形成一个复杂的“多模型协作”体系。这种协作模式通常遵循一个“规划-审计-执行”的生命周期。首先，系统会生成一个详细的执行计划（Workflow Plan），用户可以在此阶段进行审查（Audit），了解将要使用的模型、预期的步骤以及可能消耗的资源，从而确保过程的可控性。一旦用户批准，工作流便进入执行阶段，并支持大规模的并发处理，这意味着可以同时启动大量的子智能体实例来并行处理任务，极大地提高了处理效率，尤其是在面对大规模 Bug 分析、代码审查或存储库导航等复杂场景时。

### 技术细节

Agent 工作方式的关键技术体现在其工作流生命周期管理和多智能体协作模式上。在工作流生命周期中，Claude Code 首先进行“规划”，生成执行计划和流程图，而非立即执行，这为用户提供了宝贵的“审计”机会，可以审查工作流脚本、步骤、使用的模型以及预计消耗，确保过程符合预期且资源可控。一旦审计通过，便进入“执行”阶段，该阶段支持大规模并发，能够同时启动数十个甚至上百个智能体实例来处理不同的任务点。

在多智能体协作模式下，工作流通常包含几个关键阶段：
1.  **数据拉取：** 这一阶段主要负责抓取任务所需的数据，如 GitHub Issue 或代码元数据，通常不消耗智能体配额。
2.  **并行分析：** 为每个独立的任务点（例如，每一个 Bug Issue）派生出独立的子智能体进行分析。
3.  **对抗验证：** 通过让不同的智能体或模型对分析结果进行交叉比对，以提高结果的准确性和鲁棒性。
4.  **最终合成：** 由一个核心智能体负责汇总所有子智能体的分析结果和反馈，生成最终的报告或解决方案。

这种模式允许 Claude Code 灵活地调用不同的模型（如 [[Opus]]、[[Sonnet]] 或 [[Haiku]]）来发挥各自的优势，实现 [[互补优势]]，并可能通过 [[Codex CLI 交叉验证]] 等方式来增强结果的可靠性。

### 应用场景

Agent 工作方式在处理复杂软件工程任务时展现出巨大的潜力。例如，在进行大规模 Bug 分析时，它可以自动识别并派生出多个子智能体来分析不同的 Bug，并利用 [[CodeGraph MCP]] 等工具进行代码层面的深入挖掘。在代码审查场景下，Agent 工作方式能够协调多个智能体对代码进行多角度的审查，包括功能逻辑、安全性和性能等方面，并可能通过 [[三方协同]] 的方式来整合不同智能体的意见。此外，在存储库导航和理解方面，Agent 工作方式能够帮助开发者更高效地理解代码库的结构和依赖关系，实现 [[代码导航]] 和 [[代码跳转]]。这种工作方式特别适合处理需要大量数据处理、多步骤推理和跨模型协作的任务，能够显著提升开发效率和代码质量。
