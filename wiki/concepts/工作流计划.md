## 工作流计划

工作流计划（Workflow Plan）是 Claude Code 动态工作流（Dynamic Workflows）功能中的一个关键概念，它代表了系统在执行复杂任务前所生成的详细执行蓝图。与传统的即时执行模式不同，动态工作流首先会生成一个完整的执行计划和流程图，而不会立即开始执行。这个计划详细阐述了任务将被如何分解、哪些 [[子智能体]] 将被派生、它们将如何协作、需要集成哪些 [[外部工具]]（如 [[CodeGraph MCP]] 和 [[Codex CLI]]），以及预计会消耗多少计算资源（如 [[智能体配额]]）。

### 技术细节

工作流计划的生成是动态工作流生命周期的第一步，紧随用户输入激活指令（如 `ultra code`）之后。该计划通常包含以下要素：

*   **执行路径：** 描述了任务从开始到结束的整个流程，包括各个阶段的顺序和依赖关系。
*   **智能体分配：** 指定了每个任务节点或子任务将由哪个模型（如 [[Opus]]、[[Sonnet]] 或 [[Haiku]]）或类型的智能体来处理。
*   **工具集成：** 列出了在执行过程中需要调用的外部工具及其用途。
*   **资源预估：** 提供了对预计消耗的智能体配额、计算时间和潜在成本的估算。
*   **脚本结构：** 工作流计划本身可以被视为一种脚本，定义了智能体之间的通信协议和数据流。

在生成工作流计划后，系统会进入 [[审计]] 阶段，允许用户审查该计划的详细内容。用户可以检查工作流脚本、具体的执行步骤、所使用的模型以及预期的资源消耗。这一审查过程至关重要，因为它赋予了用户对自动化过程的控制权，确保了计划符合预期，并允许在执行前进行必要的修改或优化，例如调整 [[Agent 数量]] 或更换 [[模型类型]]。只有在用户批准后，系统才会进入 [[执行]] 阶段，并支持 [[大规模并发]] 执行。

### 应用场景

工作流计划在处理大规模、复杂的软件工程任务时尤为重要，例如：

*   **大规模 Bug 分析：** 在处理大量 [[GitHub Issue]] 时，工作流计划可以规划如何并行地将每个 Issue 分配给独立的智能体进行分析，然后由一个核心智能体进行 [[最终合成]]。
*   **代码审查：** 计划可以定义一个多阶段的审查流程，包括初步的代码静态分析、安全漏洞扫描，以及不同模型之间的 [[对抗验证]]，以确保审查的全面性和准确性。
*   **存储库导航与理解：** 对于大型代码库，工作流计划可以规划如何逐步构建 [[代码库理解]]，例如先进行 [[数据拉取]] 获取代码元数据，再进行 [[并行分析]]，最后通过 [[关联分析]] 来理解代码之间的依赖关系。

通过 [[工作流计划]]，Claude Code 能够从一个简单的 AI 助手转变为一个“[[智能体指挥官]]”，能够自主地规划和协调多个智能体及外部工具来高效解决复杂问题，同时提供用户对整个过程的可见性和控制力。
