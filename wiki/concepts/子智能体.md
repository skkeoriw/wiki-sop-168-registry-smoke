## 概念定义

子智能体（Sub-agent）是 Claude Code 动态工作流（Dynamic Workflows）中的核心组成部分，代表着从单一 AI 助手向“智能体指挥官”形态转变的关键技术体现。在动态工作流的架构下，一个主智能体（或称为“指挥官”）能够根据预设的“工作流计划”（Workflow Plan），自动规划、派生出多个功能各异的子智能体。这些子智能体被设计为能够独立执行特定的任务片段，例如对某个 GitHub Issue 进行初步分析、审查一段代码、或者导航代码库中的特定模块。这种设计允许工作流以高度并行化的方式运行，极大地提高了处理大规模、复杂任务的效率。子智能体之间可以进行“对抗验证”（Adversarial Validation），即通过不同的子智能体或模型对同一任务进行独立分析，然后将结果进行交叉比对，以提高最终输出的准确性和鲁棒性。最终，一个核心智能体负责汇总所有子智能体的分析结果和反馈，进行“最终合成”（Final Synthesis），生成一份全面、可靠的报告或解决方案。这种多智能体协作模式是 Claude Code 解决大规模 Bug 分析、代码审查和存储库导航等高级软件工程任务的关键。

## 技术细节

子智能体的派生和管理是动态工作流生命周期管理（Workflow Lifecycle Management）的关键环节。当用户输入 `ultra code` 指令激活动态工作流后，系统首先会生成一个“执行计划”（Workflow Plan），其中详细描述了需要执行的任务步骤以及每个步骤可能需要派生的子智能体。在“审计”（Audit）阶段，用户可以审查这个计划，包括子智能体的数量、使用的模型（如 [[Opus]]、[[Sonnet]]、[[Haiku]] 等）、以及预期的资源消耗。

在“执行”（Execution）阶段，系统能够大规模并发地启动多个子智能体实例。例如，在处理 50 个 GitHub Issue 时，系统可以同时启动数十个子智能体，每个子智能体负责分析一个或一组 Issue。这些子智能体可能被配置为使用不同的模型或采用不同的分析策略，以实现“多模型协作”（Multi-model Collaboration）和“对抗验证”。例如，一个子智能体可能使用 [[Opus]] 进行深度代码逻辑分析，而另一个子智能体则使用 [[Sonnet]] 进行代码风格检查。

子智能体还可以集成“外部工具”（External Tools），如 [[CodeGraph MCP]] 和 [[Codex CLI]]。例如，一个子智能体可能负责调用 [[CodeGraph MCP]] 来获取代码的结构信息，另一个子智能体则可能使用 [[Codex CLI]] 来执行代码测试或静态分析。这种集成使得子智能体能够访问和利用更广泛的计算资源和分析能力。

## 应用场景

子智能体的应用场景极为广泛，尤其是在处理大规模、高复杂度的软件工程任务时，其优势尤为突出：

*   **大规模 Bug 分析：** 在面对海量 Bug 报告时，可以派生大量子智能体，每个子智能体负责分析一个或一组 Bug。通过并行分析和对抗验证，可以快速定位核心 Bug，并生成详细的分析报告。
*   **代码审查：** 对于大型代码库的审查，可以为每个模块或每个 Pull Request 派生独立的子智能体进行审查。子智能体之间可以相互协作，发现潜在的 Bug、安全漏洞或代码异味。
*   **存储库导航与理解：** 子智能体可以被用于理解复杂的代码库结构，进行代码导航，查找特定函数或变量的定义和使用，以及分析代码之间的依赖关系。
*   **自动化测试与验证：** 子智能体可以被配置为执行自动化测试用例，并对测试结果进行分析和验证，从而加速软件的质量保证过程。
*   **成本与性能优化：** 通过灵活配置子智能体的数量、使用的模型以及集成工具，可以实现“性价比最大化”，在满足分析深度和准确性的前提下，优化资源消耗。

通过子智能体机制，Claude Code 能够从一个简单的代码助手，演变为一个能够自主规划、分派任务、协同工作的“智能体指挥官”，极大地提升了处理复杂软件工程问题的效率和能力。
