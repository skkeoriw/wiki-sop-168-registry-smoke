## 概念定义

消耗（Consumption）在 Claude Code 动态工作流的语境下，特指执行工作流过程中所产生的各项成本。这包括但不限于智能体（Agent）的运行时间、模型（如 Opus, Sonnet, Haiku）的调用次数和参数量、外部工具的调用以及其他计算资源的使用。动态工作流的设计理念之一便是提供透明度和可控性，尤其是在成本方面。通过其“规划-审计-执行”的生命周期管理模式，用户可以在实际执行前对工作流的预期消耗进行详细审查。这使得开发者和用户能够预估并管理项目成本，避免不必要的资源浪费，并根据预算选择最经济高效的模型组合或工作流策略。例如，在进行大规模 GitHub Issue 分析时，系统会明确列出预计的智能体配额使用量和模型调用成本，用户可以据此调整策略，例如优先使用性价比更高的模型（如 Haiku）进行初步筛选，再将关键任务交给更强大的模型（如 Opus）。这种精细化的成本管理能力是动态工作流在处理复杂、大规模任务时的一大优势。

## 技术细节

在动态工作流的“审计”阶段，系统会生成一份详细的执行计划，其中明确列出了预期的智能体数量、模型调用类型（例如，是调用 Opus 进行代码生成，还是 Sonnet 进行文本摘要）、每次调用的参数量以及预计的“消耗”总额。这通常以“智能体配额”或具体的成本估算形式呈现。工作流的并行执行能力，例如在处理 50 个 Issue 时同时启动数十个智能体实例，虽然提高了效率，但也可能显著增加消耗。因此，审计阶段的消耗预估至关重要，它允许用户在批准执行前进行干预，例如：

*   **模型选择优化：** 根据任务的复杂度和对精度的要求，选择不同层级的模型。例如，对于简单的代码格式检查，可以使用成本较低的 Haiku；而对于复杂的逻辑分析或核心 Bug 识别，则可能需要 Opus。
*   **智能体数量调整：** 限制并行执行的智能体数量，以控制并发带来的成本增长。
*   **工作流脚本审查：** 检查工作流脚本中是否存在不必要的循环、冗余的调用或低效的逻辑，这些都可能导致不必要的消耗。
*   **预算控制：** 设定一个最大消耗阈值，一旦预估消耗接近或超过该阈值，系统可以自动暂停或拒绝执行。

## 应用场景

*   **大规模 GitHub Issue 分析与验证：** 在处理海量 GitHub Issue 时，动态工作流能够预估分析和验证这些 Issue 所需的智能体配额和模型调用成本。用户可以根据预估消耗，决定是分批处理，还是调整模型组合以降低成本。例如，可以先用 Sonnet 进行初步分类和信息提取，再用 Opus 对高优先级 Issue 进行深入分析。
*   **CodeGraph MCP 集成与代码导航：** 集成 CodeGraph MCP 后，工作流可能需要调用多个模型来理解代码库的上下文。消耗的预估可以帮助开发者了解，进行一次全面的代码导航或关联分析需要多少资源，从而在成本和效率之间找到平衡点。
*   **成本与性能优化：** 用户可以利用消耗预估功能，对比不同模型混合策略（如 [[多模型协作]]）或不同 [[智能体]] 配置下的成本差异，选择性价比最高的方案。例如，通过 [[Codex CLI]] 进行交叉验证时，可以比较使用不同模型组合进行验证的消耗和准确性。
