## 概念定义

Agent 数量（Agent Count）在动态工作流（Dynamic Workflows）的语境下，指的是在执行一个复杂任务时，系统内部调用的、具备特定功能的独立智能体（Agent）的总数。这些 Agent 可以是单一大型语言模型（LLM）的不同实例，也可以是不同模型系列（如 [[Opus]]、[[Sonnet]]、[[Haiku]]）的组合，甚至是集成 [[第三方 CLI 工具]] 的封装。每个 Agent 都被设计用来执行工作流中的一个或多个特定阶段（Stage）的任务，例如数据拉取、代码分析、报告合成、对抗验证等。

在 [[Claude Code Dynamic Workflows]] 中，Agent 的数量并非固定不变，而是根据任务的复杂度和用户设定的策略动态调整。例如，一个简单的代码审查任务可能只需要少数几个 Agent，而一个需要深度系统级架构分析的任务则可能需要数十甚至上百个 Agent 来协同工作，每个 Agent 负责处理一部分代码库导航、关联分析或 [[潜在 Bug]] 的识别。

理解和控制 Agent 数量对于实现 [[性价比]] 最大化至关重要。过多的 Agent 可能导致 [[资源失控]]、[[消耗]] 过大以及不必要的 [[并发执行]] 开销，而过少的 Agent 则可能无法充分利用 [[互补优势]]，导致任务执行效率低下或分析深度不足。因此，通过 [[Plan 预览]] 功能，用户可以明确要求展示“每个阶段预计使用的 Agent 数量”和“模型类型”，从而在任务启动前进行有效的成本和性能评估，并根据需要进行调整。这体现了动态工作流在 [[灵活配置]] 和 [[成本与性能优化]] 方面的核心优势。

## 技术细节

Agent 数量的动态调整是 [[动态工作流]] 引擎的核心能力之一。引擎会根据任务的输入、预设的 [[工作流关键词]] 以及用户指定的策略来规划 Agent 的调用序列和数量。

*   **任务分解与 Agent 分配：** 复杂任务会被分解为一系列更小的子任务，每个子任务会被分配给一个或多个具备相应能力的 Agent。例如，代码审查任务可能首先由一个 Agent 进行初步的 [[代码审查]]，然后将发现的问题传递给另一个 Agent 进行 [[T10 级别 Bug]] 的深度分析。
*   **模型混合策略：** 为了优化成本和性能，系统会根据任务阶段采用不同的模型。例如，在分析和验证的中间阶段，可能会优先使用成本较低的 [[Haiku]] 或 [[Sonnet]] 模型来处理大量数据，而在最初的规划和最终的 [[最终合成]] 阶段，则会调用更强大的 [[Opus]] 模型来确保输出的准确性和权威性。这种策略直接影响了每个阶段所需要的 Agent 类型和数量。
*   **[[Plan 预览]] 与用户干预：** 用户可以通过强制开启 [[Plan 预览]] 来审查系统生成的执行计划，其中会明确列出每个阶段预期的 [[Agent 数量]] 和 [[模型类型]]。用户可以在此阶段进行调整，例如减少某些阶段的 Agent 数量以控制 [[预算控制]]，或者增加 Agent 来提升 [[分析深度]]。
*   **[[脚本复用]] 与 Agent 驱动：** 动态工作流生成的 `.js` 脚本详细记录了 Agent 的调用逻辑和与 [[外部工具]] 的交互方式。开发者可以通过审查这些脚本来理解 Agent 是如何驱动 [[Codex CLI]] 等工具的，并可以将这些脚本保存下来，用于其他项目或进行定制化修改，从而实现 Agent 的高效复用。
*   **[[MCP 工具]] 集成：** 结合 [[CodeGraph MCP]] 等 [[MCP 工具]]，可以将 Claude Code 的能力从简单的对话扩展到复杂的 [[系统级架构分析]]。这种集成通常意味着需要调用更多、更专业的 Agent 来处理代码元数据、依赖关系分析等任务，从而显著增加 Agent 的总数。

## 应用场景

Agent 数量的灵活配置在多种场景下都至关重要：

*   **大规模代码库分析：** 当需要对包含数百万行代码的庞大代码库进行分析时，系统需要调用大量的 Agent 来并行处理不同的代码模块、进行 [[存储库导航]]、[[代码导航]] 和 [[关联分析]]。合理控制 Agent 数量可以平衡分析速度和计算资源消耗。
*   **[[GitHub Issue 分析]] 与 [[崩溃数据]] 处理：** 分析大量的 GitHub Issue 或崩溃日志时，需要多个 Agent 来进行 [[自动化筛选]]、[[优先级排序]] 和 [[核心 Bug]] 的识别。Agent 数量的多少直接影响了处理效率和发现关键问题的能力。
*   **[[对抗验证]] 与 [[交叉比对]]：** 在进行 [[对抗验证]] 或 [[交叉比对]] 时，可能需要部署多个 Agent，每个 Agent 使用不同的模型或策略来验证同一份数据或代码。增加 Agent 数量可以提高验证的鲁棒性和准确性。
*   **[[三方协同]] 与 [[外部工具集成]]：** 当工作流需要集成 [[Codex CLI]]、[[CodeGraph MCP]] 等 [[第三方 CLI 工具]] 时，每个工具的调用和交互都需要由专门的 Agent 来负责。Agent 数量的增加反映了工作流对外部能力的依赖程度。
*   **[[预算控制]] 与 [[性价比]] 优化：** 在资源有限的情况下，用户可以通过调整 Agent 数量来控制 [[消耗]]。例如，在非关键任务中减少 Agent 数量，或者在非高峰时段执行任务，以实现 [[性价比]] 的最大化。

通过对 Agent 数量的精细化管理，用户可以更好地驾驭 [[动态工作流]] 的强大能力，实现从简单对话到复杂系统级分析的平滑过渡。
