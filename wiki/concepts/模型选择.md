## 模型选择

模型选择是 [[Claude Code Dynamic Workflows]] 功能中的一个关键决策点，尤其是在 [[多模型协作]] 和 [[智能体指挥官]] 模式下。它涉及到根据当前任务的性质、复杂性、对性能的要求以及成本预算，从一系列可用的 AI 模型（例如 Claude 的 Opus、Sonnet、Haiku 等，或潜在的 [[非 Claude 系列模型]]）中，动态地选择最适合执行特定子任务的模型。这种选择并非随机，而是基于对不同模型能力、成本和速度的深入理解，以实现 [[性价比最大化]] 和任务效率的优化。

### 技术细节

在 [[动态工作流]] 的 [[工作流生命周期]] 中，模型选择通常发生在 [[规划 (Workflow Plan)]] 阶段，并在 [[审计 (Audit)]] 阶段向用户展示。工作流引擎会分析任务的类型（例如，是需要深度代码逻辑分析、快速 Bug 报告生成，还是大规模数据抓取），并根据预设的策略或用户指定的偏好，为每个 [[子智能体]] 或 [[任务点]] 分配最合适的模型。例如，对于需要高度推理能力和复杂代码理解的任务，可能会选择 Opus 这样的旗舰级模型；而对于更常规的分析或数据提取任务，则可能选用 Sonnet 或 Haiku 以降低成本和提高速度。这种选择过程可能涉及对模型上下文窗口大小、推理速度、[[代码逻辑敏感度]]、[[分析深度]] 以及 [[成本与性能优化]] 的权衡。工作流脚本中可以明确指定模型，或者依赖于系统内置的智能选择机制。

### 应用场景

模型选择在多种高级应用场景中至关重要：

*   **大规模 Bug 分析：** 在处理大量 [[GitHub Issue]] 时，可以将需要深度分析和根源定位的任务分配给 Opus，而将初步的 Issue 分类和信息提取任务分配给 Sonnet 或 Haiku，从而平衡分析深度和处理速度。
*   **代码审查：** 对于关键模块或复杂逻辑的审查，可以选择 Opus 以确保最高级别的准确性和 [[权威性]]；对于常规代码风格检查或简单 Bug 的发现，则可使用更经济的模型。
*   **存储库导航与代码理解：** 在需要快速扫描大量代码以理解整体结构或查找特定信息时，可以优先使用速度更快的模型，而在需要深入理解特定代码段的逻辑时，再切换到更强大的模型。
*   **成本与性能优化：** 用户可以通过模型选择来控制 [[预算控制]] 和 [[消耗]]，确保在满足任务需求的前提下，最小化 [[资源浪费]]。例如，在开发早期阶段，可能更侧重于速度和成本，而在产品发布前，则会倾向于使用更强大的模型来确保质量。

通过精细化的模型选择，[[Claude Code Dynamic Workflows]] 能够实现更高效、更具成本效益的 [[多智能体协作]]，将 AI 助手从单一工具转变为能够根据任务动态调整策略的 [[智能体指挥官]]。
