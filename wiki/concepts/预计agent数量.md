## 概念定义

预计 Agent 数量（Estimated Agent Count）是 Claude Code 动态工作流（Dynamic Workflows）功能在执行复杂任务时，系统根据任务的规模、复杂度和并行处理需求，预先规划和启动的智能体（Agent）实例数量的估算值。在动态工作流的“规划”（Workflow Plan）阶段，系统会分析待处理的任务项，例如大量的 GitHub Issue 或代码审查请求，并据此推断出需要多少个独立的子智能体（Sub-agents）来并行处理这些任务，以达到最高效率。这个预估值不仅指导了系统资源的分配，也为用户提供了对工作流执行规模和潜在消耗的初步认知。例如，当处理 50 个 Issue 时，系统可能会预估启动数十个 Agent 实例以实现大规模并发执行。这个概念强调了动态工作流在资源管理和任务分解方面的智能化能力，旨在通过精细化的 Agent 分配来优化处理速度和资源利用率。

## 技术细节

在 Claude Code 的动态工作流中，预计 Agent 数量的生成是基于对输入任务的分析。当用户触发 `ultra code` 指令后，系统会进入“规划”阶段。在此阶段，Claude Code 会评估任务的粒度（例如，每个 Issue 的独立性）和所需的处理深度。如果任务可以被分解为多个可以并行处理的独立单元，系统就会计算出所需的 Agent 数量。这个计算过程可能涉及到：

*   **任务分解算法：** 将大型任务分解为更小的、可管理的子任务。
*   **并行处理能力评估：** 考虑系统当前可用的计算资源和模型处理能力。
*   **模型协作策略：** 如果任务需要不同模型（如 [[Opus]]、[[Sonnet]]、[[Haiku]]）的协同，也会影响 Agent 的数量和类型。
*   **外部工具集成：** 与 [[CodeGraph MCP]] 或 [[Codex CLI]] 等外部工具的交互也可能影响 Agent 的分配。

在“审计”（Audit）阶段，用户可以审查这个“预计 Agent 数量”，以及每个 Agent 可能执行的具体任务、使用的模型以及预计的消耗。这为用户提供了对工作流执行细节的透明度，并允许在正式“执行”（Execution）前进行调整。

## 应用场景

预计 Agent 数量的概念在多种高级应用场景中至关重要：

*   **大规模 Bug 分析：** 当需要分析成百上千个 GitHub Issue 时，系统会预估需要多少个 Agent 来并行处理每个 Issue 的分析，从而快速定位核心 Bug。
*   **代码审查：** 对于大型代码库的审查，系统会根据代码模块的数量和复杂性，预估需要多少个 Agent 来执行代码审查任务，并可能进行 [[对抗验证]]。
*   **存储库导航与理解：** 在需要对整个代码库进行深入理解时，系统会预估 Agent 数量来执行代码元数据提取、依赖关系分析等任务。
*   **成本与性能优化：** 通过预估 Agent 数量，用户可以更好地控制工作流的执行成本，并根据预估值调整模型选择策略，以实现 [[性价比]] 最大化。例如，对于不那么关键的任务，可能会选择更经济的模型，并相应调整 Agent 数量。
*   **资源管理：** 预估 Agent 数量有助于防止 [[资源失控]]，确保系统不会因为启动过多 Agent 而导致资源耗尽或 [[资源浪费]]。

通过对“预计 Agent 数量”的清晰展示和控制，Claude Code 的动态工作流功能使用户能够更有效地管理和执行复杂的软件工程任务。
