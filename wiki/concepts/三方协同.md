# 三方协同

三方协同是一种高级的代码审查策略，它通过同时调用多个不同类型和能力的智能体或模型来执行同一项任务，从而实现优势互补、提高准确性并优化资源消耗。这种模式尤其适用于代码审查场景，其中可以引入 Claude 系列模型（如 Opus、Sonnet）与第三方命令行工具（如 [[Codex CLI]]）进行并行分析和交叉验证。其核心在于利用不同工具在处理特定代码逻辑、发现潜在 Bug 或评估效率方面的差异，以期获得比单一模型更全面、更可靠的分析结果。

## 技术细节

在三方协同的实践中，通常会设计一个工作流，该工作流能够并行启动多个独立的审查任务。例如，可以同时启动 [[Codex CLI]]、[[Opus]] 模型和 [[Sonnet]] 模型进行代码审查。[[Codex CLI]] 因其对特定代码逻辑的敏感度，可能能够发现 Claude 模型可能忽略的潜在 Bug，从而实现“互补优势”。这种方法不仅能提升代码质量，还能在效率上带来显著提升。实测数据显示，调用 [[Codex CLI]] 的智能体仅消耗极少量的 Token（如 20K），远低于 [[Opus]] 或 [[Sonnet]] 模型可能消耗的 100K+ Token，这体现了其在“效率对比”方面的优势。这种策略允许用户根据任务需求灵活选择模型组合，以达到“成本与性能优化”的最佳平衡。

## 应用场景

三方协同最典型的应用场景是“Codex CLI 交叉验证代码审查”。在这种模式下，工作流被设计为同时启动 [[Codex CLI]]、[[Opus]] 和 [[Sonnet]] 三个独立的审查任务。通过这种方式，可以利用 [[Codex CLI]] 在特定代码逻辑上的敏感度，发现 Claude 系列模型可能遗漏的“潜在 Bug”。同时，通过对比不同模型在审查同一段代码时的 Token 消耗，可以清晰地看到 [[Codex CLI]] 在效率上的巨大优势，从而为用户提供更具性价比的分析方案。这种“对抗验证”的模式，能够显著提升代码审查的深度和广度，确保最终产出的报告具有更高的“准确性”和“权威性”。

此外，三方协同的理念也可以扩展到其他需要多角度分析的场景，例如在 [[动态工作流]] 的不同阶段，根据任务性质（如数据抓取、初步分析、对抗验证、报告合成）选择最适合的模型组合（如 [[Haiku]]、[[Sonnet]]、[[Opus]]），以实现“海量数据处理”和“逻辑能力”的最佳结合，最终实现“性价比最大化”。

## 成本与性能优化策略

为了平衡分析深度与 Token 消耗，Dynamic Workflows 提供了灵活的配置选项：

| 阶段 | 任务性质 | 建议模型 | 优化效果 |
| :--- | :--- | :--- | :--- |
| **第一阶段** | 数据抓取 | N/A | 无 Agent 消耗 |
| **第二阶段** | 初步分析 | [[Haiku]] (H库) | 极低成本处理海量数据 |
| **第三阶段** | 对抗验证 | [[Sonnet]] (S模型) | 兼顾逻辑能力与速度 |
| **第四阶段** | 报告合成 | [[Opus]] (O模型) | 确保最终产出的准确性与权威性 |
