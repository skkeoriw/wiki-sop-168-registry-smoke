## 概念定义

动态工作流（Dynamic Workflows）是 Claude Code V2.1.162 版本中引入的一项革命性功能，它标志着 AI 助手从单一响应者向“智能体指挥官”的根本性转变。该功能的核心在于其强大的自动化能力，能够自主地规划复杂的任务流程，并在此过程中动态地派生出多个专门的子智能体（Sub-agents）。这些子智能体可以协同工作，集成外部工具（如 [[CodeGraph MCP]] 和 [[Codex CLI]]），并利用 [[多模型协作]] 的优势，以极高的效率解决大规模、多阶段的软件工程挑战。动态工作流的引入，极大地提升了 AI 在处理 Bug 分析、代码审查、存储库导航等复杂场景下的能力，使其能够处理以前需要人工大量干预的任务。

## 技术细节

动态工作流的激活通常通过特定的 [[工作流关键词]]（如 `ultra code`）来实现。一旦激活，工作流便进入一个清晰的生命周期管理阶段，遵循“规划-审计-执行”的路径。

1.  **规划 (Workflow Plan)：** 系统首先会生成一个详细的执行计划和流程图，明确任务的分解、子智能体的分配以及预期的步骤。这一阶段的关键在于，系统不会立即执行，而是提供一个可供审查的蓝图。
2.  **审计 (Audit)：** 用户在此阶段可以对生成的 [[工作流脚本]]、每一个执行步骤、所使用的模型（如 [[Opus]]、[[Sonnet]]、[[Haiku]] 等）以及预期的资源消耗（如智能体配额、计算成本）进行详细审查。这一环节确保了透明度和可控性，允许用户在执行前进行调整和优化。
3.  **执行 (Execution)：** 经过用户审计和批准后，工作流进入执行阶段。动态工作流支持 [[大规模并发]] 执行，这意味着系统可以同时启动数十甚至上百个子智能体实例来处理不同的任务点，例如同时分析 50 个 GitHub Issue。

在多智能体协作模式下，工作流通常包含以下阶段：

*   **数据拉取：** 初始阶段，系统会抓取任务所需的数据，如 Issue 或代码元数据。这一步通常不消耗智能体配额。
*   **并行分析：** 为每个独立的任务点（如每个 Issue）派生出 [[独立智能体]] 进行并行分析。
*   **对抗验证：** 不同的智能体或模型会对分析结果进行 [[交叉比对]]，以实现 [[对抗验证]]，提高结果的鲁棒性和准确性。
*   **最终合成：** 由一个 [[核心智能体]] 负责汇总所有子智能体的分析结果和反馈，生成最终的报告或解决方案。

此外，动态工作流还强调 [[外部工具集成]]，能够无缝对接 [[CodeGraph MCP]] 和 [[Codex CLI]] 等工具，扩展 AI 的能力边界。

## 应用场景

动态工作流在软件工程领域具有广泛的应用前景，尤其是在处理大规模、高复杂度的任务时：

*   **大规模 Bug 分析：** 能够同时分析大量的 GitHub Issue，识别 [[核心 Bug]]，并提供详细的修复建议。通过 [[并行分析]] 和 [[对抗验证]]，可以显著提高 Bug 检测的准确性和效率。
*   **代码审查：** 动态工作流可以自动化代码审查流程，派生多个智能体对代码进行不同维度的审查，例如检查代码风格、潜在的安全漏洞、性能瓶颈等，并进行 [[三方协同]]，最终由一个核心智能体汇总审查意见。
*   **存储库导航与理解：** 对于大型代码库，动态工作流可以帮助开发者快速导航、理解代码结构、分析模块间的依赖关系，并进行 [[关联分析]]，极大地缩短了新成员的上手时间和代码库的维护成本。
*   **自动化测试与回归：** 动态工作流可以被设计来自动化生成测试用例，执行回归测试，并分析测试结果，从而加速软件的发布周期。
*   **复杂系统诊断：** 在生产环境中，当出现复杂的故障时，动态工作流可以整合来自不同监控系统的数据，并调用各种诊断工具，快速定位问题根源。

通过 [[模型混合策略]]，动态工作流能够根据任务需求灵活选择最适合的模型组合，实现 [[性价比最大化]]，例如使用 [[Haiku]] 进行初步筛选，再由 [[Opus]] 进行深度分析。
